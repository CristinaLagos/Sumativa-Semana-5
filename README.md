
Este proyecto tiene como objetivo diseñar, normalizar y transformar un Modelo Entidad–Relación (MER) incompleto en un Modelo Entidad–Relación Extendido (MER-E) y posteriormente en un Modelo Relacional para el caso de estudio Retail Solari S.A., utilizando la herramienta Oracle SQL Developer Data Modeler. El trabajo se desarrolla a partir del análisis del contexto de negocio y de las reglas definidas en la rúbrica de evaluación, asegurando que el modelo resultante represente correctamente las operaciones de la empresa.

Durante el análisis se identificaron las entidades principales del sistema, junto con sus atributos identificadores, obligatorios y opcionales. Asimismo, se definieron las relaciones entre las entidades, considerando las restricciones de cardinalidad, opcionalidad y transferibilidad establecidas por las reglas de negocio. Se modeló la estructura geográfica mediante las entidades REGIÓN, COMUNA y SUCURSAL, permitiendo reflejar la distribución de las sucursales a nivel nacional.

Los proveedores fueron modelados utilizando un enfoque de supertipo y subtipos, donde PROVEEDOR actúa como supertipo y PERSONA_NATURAL y EMPRESA como subtipos, diferenciados mediante un atributo discriminador. Esta especialización permite registrar adecuadamente la información específica de cada tipo de proveedor, cumpliendo con los requerimientos del caso.

En cuanto a la gestión de productos, se definió una jerarquía clara entre MARCA, MODELO y PRODUCTO. La relación entre MARCA y MODELO es identificadora y no transferible, ya que cada modelo pertenece a una única marca y su identificador incluye la marca que lo fabrica. El PRODUCTO se define como una variante concreta de un modelo, evitando la redundancia de datos y asegurando el cumplimiento de la tercera forma normal. Además, se incorporó la entidad CATEGORIA para normalizar la clasificación de los productos.

El modelo también contempla la gestión de stock mediante la entidad BODEGA y el registro de ventas a través de las entidades CLIENTE, BOLETA y DETALLE_BOLETA, permitiendo almacenar la información necesaria para representar las transacciones comerciales de la empresa de forma consistente y estructurada.

Finalmente, el modelo fue normalizado aplicando las tres primeras formas normales, eliminando dependencias parciales y transitivas, y luego transformado a un Modelo Relacional. Como resultado, se definieron las tablas, claves primarias, claves foráneas y restricciones de integridad correspondientes, y se generó un script DDL en Oracle SQL que permite la creación del modelo físico conforme a los requerimientos establecidos en la rúbrica.
